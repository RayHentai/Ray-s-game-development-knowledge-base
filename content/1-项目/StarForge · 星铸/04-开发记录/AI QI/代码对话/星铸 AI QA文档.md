## 关于第一点：为什么这样做

你说得对，我跳过了"为什么"直接给了"怎么做"，这不是好的教学。补上来。

**为什么要 `Editor` 文件夹 + `ProjectSetup.cs`？**

Unity 有一个规则：放在 `Editor` 文件夹里的脚本只在编辑器里运行，不会打包进游戏。`ProjectSetup.cs` 是个一次性工具，用完就可以忘掉它，所以放这里正合适。这也是 Unity 项目的通用惯例，以后你看别人的项目也会看到这个文件夹。

**为什么 `ResourceManager` 要用单例（Singleton）？**

游戏里任何地方都可能需要"加矿石"——点击触发、怪物掉落、任务奖励……如果每个脚本都要找到 `ResourceManager` 这个对象再调用，会很麻烦。单例的意思是：全局只有一个，任何地方写 `ResourceManager.Instance.Add(...)` 就能直接用，不需要先找到它。代价是它会一直活在内存里，但对这种全局管理器来说完全合理。

**为什么用 `事件（event）` 而不是让 UI 每帧去查数字？**

两种方式都能工作，但每帧查询（`Update` 里 `GetResource("ore")`）是在不断地问"有没有变化"——哪怕什么都没发生。事件是反过来的："有变化了我通知你"。游戏越复杂，这个区别越重要。这个模式叫**观察者模式**，是游戏开发中最常用的设计模式之一。

**为什么 `ClickCollector` 用射线检测而不是 `OnMouseDown`？**

`OnMouseDown` 更简单，但 Unity 的新版本对它的支持越来越弱，在 WebGL 和移动端有兼容问题。射线检测是标准做法，而且能精确控制"点击到底打中了什么"，以后做多层物体叠加时不会出错。

---

## 关于第二点：GDD 没有定死，它是草稿

这个要说清楚：**GDD 永远是活文档，不是合同。**

我帮你生成的那份 GDD 是基于你描述的参考游戏推导出来的一个"合理起点"，目的是让你有个地方开始，而不是把你锁死。它的每一行都可以改、删、推翻。

专业团队的 GDD 也是这样——Valve 做 Half-Life 的时候，最终游戏和最初设计文档差了十万八千里，这很正常，叫做**迭代设计**。

## 这一步要学的核心概念：ScriptableObject

在写代码之前，先讲一个你以后会反复用到的 Unity 概念——**ScriptableObject**。

**问题场景：** 你的游戏最终会有十几种建筑，每种建筑有名字、费用、产出、图标……如果每种建筑都硬写在代码里，改一个数字就要重新编译，而且策划（就是你自己）没法直接在 Unity Inspector 里调数值。

**ScriptableObject 的作用：** 把"数据"从"逻辑"里分离出来。你在 Inspector 里创建一个"采集器配置文件"，填写费用=10、产出=1/s，代码只负责读取这个配置文件来运行。想改数值？直接在 Inspector 里改，不用动代码。

这个模式在专业 Unity 项目里几乎无处不在，现在就开始用。

## 这次升级解决了什么问题

旧版的问题根源是**数据和代码耦合**：

```csharp
// 旧版：加一种资源就要改一次UI代码
case "ore":    UpdateText(oreText, ...); break;
case "crystal": UpdateText(crystalText, ...); break;
// 每次改资源ID，UI就坏掉
```

新版的思路：**UI 不关心有哪些资源，只负责显示 ResourceManager 告诉它的内容**。以后你加了新矿物、改了ID，UI 自动跟上，不需要动这个文件。这个思路叫**数据驱动UI**，是做系统化游戏最常用的模式。

**commit 信息怎么写？** 用这个格式：

|前缀|含义|例子|
|---|---|---|

|`fix:`|修bug|`fix: 修复资源ID不匹配问题`|

这套规范叫 **Conventional Commits**，是业界标准，写在作品集里很加分。

## 刚才发生了什么（重要）

你点的那个按钮是 Cursor 内置 Git 面板的**"放弃所有更改"**，它的作用是：把所有**未提交的修改**直接丢掉，回到上一次 `commit` 的状态。

因为你还没有执行过第一次 `git commit`，所以 Git 眼里"上一次提交"根本不存在，所有文件都被当成"未追踪的新文件"直接删掉了。

这也是为什么我说 **Git 配置必须今天做** —— 不是为了以后，是为了保护现在。

提交完之后，那个"放弃所有更改"按钮就算点了也只会回到这次提交，不会删文件。

**第二件：把 Git GUI 按钮的危险区域认识清楚**

Cursor/VSCode 的 Git 面板里有几个危险按钮，以后看到要谨慎：

```
🔄 放弃所有更改   ← 刚才踩的坑，慎点
🗑️ 丢弃此文件     ← 针对单个文件，同样危险
✅ 暂存所有更改   ← 安全，相当于 git add .
💾 提交           ← 安全，相当于 git commit
```

**第三件：以后养成这个习惯**

```

每次写完一个功能  →  马上 git add . && git commit
```

## 污染系统：完全可行，而且和现有系统咬合得很好

污染本质上是一个**反向资源**——其他资源你想要越多越好，污染你想要越少越好。如果把它设计成一个全局数值，实现成本很低，但带来的游戏深度很高。

它可以和你现有的三个系统自然联动：

```
燃料机器运行  →  产生污染
工厂生产线    →  产生污染
污染值过高    →  怪物变强/刷新加快（战斗系统）
污染值过高    →  采集效率下降（资源系统）
净化建筑      →  消耗电力降低污染（电力系统）
```

这条链条本身就构成了一个**决策压力**：你可以烧燃料快速发展，但污染会让你越来越难打，逼你去升级清洁能源。这是非常优秀的游戏设计，有点像《异星工厂》里污染吸引虫群的机制。

**MVP 里只需要做三件事：**

- 一个全局 `pollutionLevel` 数值（0-100）
- 部分建筑运行时每秒 +pollution
- 超过阈值触发警告 + 怪物强化

实现成本大约 **4-5小时**，完全在范围内。

## ResourceManager 架构讲解

**它是什么**

整个游戏的资源数据中心，所有和资源有关的读写都必须经过它。其他脚本不能自己存资源数字，只能调用 ResourceManager 的方法。这叫**单一数据源**，好处是你永远知道资源数字在哪里，不会出现"A脚本显示50，B脚本显示30，实际是多少？"这种混乱。

**内部结构**

```
ResourceManager
├── ResourceData（数据结构）
│   每种资源的所有属性打包在一起：
│   id、displayName、tier、amount、perSecond、
│   hasDeposit（有无矿藏）、depositAmount（矿藏量）、
│   isFuel（是否可燃）、heatValue（热值）、isRenewable（可再生）
│
├── resources（List）
│   所有资源的列表，InitResources() 里定义了地球全部19种资源
│
├── resourceDict（Dictionary）
│   id → ResourceData 的快速查找表
│   有了它，Get("ore_iron") 比遍历列表快得多
│
└── 两个事件
    OnResourceChanged   资源数量变化时广播（UI监听）
    OnDepositChanged    矿藏储量变化时广播（矿脉UI监听）
```

**七个公开方法，记住这七个就够了**

|方法|谁会调用|做什么|
|---|---|---|
|`Add(id, amount)`|ClickCollector、怪物掉落|增加资源，不超过上限|
|`Spend(id, amount)`|BuildingManager、科技树|消耗资源，不够返回false|
|`Get(id)`|任何需要查询的地方|查当前数量|
|`HasEnough(id, amount)`|BuildingManager的CanBuild|够不够，返回true/false|
|`AddPerSecond(id, rate)`|BuildingManager建造时|注册自动产出速率|
|`RemovePerSecond(id, rate)`|BuildingManager拆除时|注销自动产出速率|
|`ApplyDifficultyMultiplier(x)`|游戏开始时GameManager调用|按难度系数缩放全部矿藏量|

**Update 里做了什么**

每帧检查所有有 `perSecond > 0` 的资源，按时间累加。有矿藏限制的资源会同时扣减矿藏储量，矿藏耗尽后自动把 `perSecond` 清零，采矿机停止产出。

## 架构说明

**三个脚本的职责分工：**

```
RecipeData（ScriptableObject）
→ 纯数据：这个配方需要什么、产出什么、在哪台机器跑
→ 不包含任何逻辑，只是一个配置文件

ProcessingBuilding（MonoBehaviour）
→ 纯逻辑：机器怎么运转、计时、产出、副产物
→ 不关心UI长什么样，只管数据和状态

MachineUI（MonoBehaviour）
→ 纯表现：把 ProcessingBuilding 的状态显示出来
→ 接收玩家输入，告诉 ProcessingBuilding 该做什么
```

**用到的设计模式：**

**状态机（State Machine）**：机器有四个状态——关机/空闲/加工中/输出满了。每帧只需判断当前状态该做什么，逻辑清晰不混乱。比一堆 `if/else bool` 好维护得多。

**数据驱动（Data-Driven）**：ProcessingBuilding 本身不知道自己在加工铁还是铜，它只读取 RecipeData 里的配置来运行。50种配方，1个脚本，这就是数据驱动的价值。

**策略模式（Strategy Pattern）**：ProcessingBuilding 把"这个配方怎么执行"委托给 RecipeData，自己只负责"什么时候执行"。切换配方就是切换策略，不改任何逻辑代码。

## 关于"方法签名"是什么

这是你问的好问题。方法签名就是方法的"接口部分"，只看名字、参数、返回值，不看内部实现。

比如 `InventoryManager` 的完整代码很长，但给另一个脚本用时只需要知道：

```csharp
// InventoryManager 方法签名
public void Add(string resourceId, float amount)
public bool Spend(string resourceId, float amount)
public float Get(string resourceId)
public bool HasEnough(string resourceId, float amount)
```

这四行就够了，不需要贴整个文件。其他脚本只调用这些方法，不关心里面怎么实现的。

---

## 关于拆散写代码会不会出问题

这是个真实的风险，但可以控制。问题的根源是**接口不一致**——A 脚本改了方法名，B 脚本还在调旧名字，就会报错（你今天遇到的 MachineUI 编译错误就是这个）。

控制方法是：**每次改脚本时，同时列出影响到的其他脚本**。我会在写代码对话框里主动告诉你"这次改动影响了哪些文件，需要同步更新哪些方法签名"。你再拿着这个清单去逐一检查或开新对话处理。

只要 GDD 里的脚本清单和依赖关系保持更新，拆散写是没问题的。

# Claude

`InventoryManager` 和 UI 的关联方式是**事件驱动**，UI 不主动查询数据，而是订阅事件等待通知。

---

## 关联方式

`InventoryManager` 有一个公开事件：

```csharp
public event Action<string, float> OnItemChanged;
```

每当背包里任何物品数量变化时，这个事件就会触发，传出两个参数：

- `string`：变化的资源 ID，比如 `"ore_iron"`
- `float`：变化后的新数量

---

## UI 脚本需要做的三件事

**第一：订阅事件**

在 UI 脚本的 `Start()` 里：

```csharp
void Start()
{
    InventoryManager.Instance.OnItemChanged += OnItemChanged;
    RefreshAll(); // 初始化时刷新一次全部显示
}
```

**第二：取消订阅（防内存泄漏）**

在 UI 脚本的 `OnDestroy()` 里：

```csharp
void OnDestroy()
{
    if (InventoryManager.Instance != null)
        InventoryManager.Instance.OnItemChanged -= OnItemChanged;
}
```

**第三：响应事件，更新显示**

```csharp
void OnItemChanged(string resourceId, float newAmount)
{
    // 只更新变化的那一行，不刷新整个列表
    // 根据 resourceId 找到对应的 UI 行，更新数字
}
```

---

## 初始化全部显示

UI 第一次显示时需要遍历所有背包物品：

```csharp
void RefreshAll()
{
    var allItems = InventoryManager.Instance.GetAllItems();
    foreach (var kvp in allItems)
    {
        // kvp.Key = resourceId
        // kvp.Value = 数量
        // 用 ResourceManager.Instance.GetDisplayName(kvp.Key) 获取中文名
    }
}
```

---

## 一句话总结

> UI 在 `Start()` 里订阅 `OnItemChanged` 事件，收到通知时更新对应行的数字；`OnDestroy()` 里取消订阅。不需要在 `Update()` 里每帧查询。
